Privacy Architecture & Technical Specs

Read how GarudaAI evaluates content locally without network leak hazards.

The "Zero Cloud" Privacy Pact

Traditional anti-phishing products inspect links or emails by sending them to web APIs. If the user is viewing confidential records, financial statements, or personal chats, this leaks sensitive information to service vendors.

"GarudaAI is built on the principle that threat inspection should never require information disclosure. All heuristics, OCR character mapping, and Transformer classification tasks are executed exclusively inside the browser environment."

How the Scanning Pipeline Operates Offline:

1. Input Hook

Characters staged strictly inside the DOM.

2. ONNX Inference

MiniLM model executes on client JS thread.

3. Local Report

Risk values calculated and loaded in view.

FAQ

Q: Is my data safe from browser extensions?

Yes. By running strictly inside standard sandboxed memory, GarudaAI doesn't transmit network traffic that background monitors could intercept. Data resides exclusively in temporary JS variables or your sandbox IndexedDB.

Q: How are new phishing patterns updated offline?

The Service Worker updates heuristics rules and local models whenever the application is opened online. Once the models are refreshed, they remain cached for fully offline verification checks.

Executive Overview
What is GarudaAI and why does it exist?

GarudaAI is a privacy-first, offline-first anti-phishing Progressive Web App (PWA) that detects phishing attempts entirely inside the user's browser — no servers, no cloud APIs, no telemetry. Every computation runs locally on the device's CPU/GPU.

🔒

Zero Telemetry

User inputs are never transmitted to any external server. All data stays inside the browser.

📴

Offline Capable

Once models are downloaded, GarudaAI can operate in fully air-gapped environments.

🤖

Local AI Inference

Uses WebGPU-accelerated LLM and WASM-compiled OCR for on-device intelligence.

The Problem

Traditional anti-phishing tools require users to upload sensitive content (messages, screenshots, links) to cloud servers — exposing private data, requiring internet connectivity, and adding latency. GarudaAI solves all three problems by moving every computation to the client.

Key Capabilities at a Glance

  • Text Scanner — Paste any suspicious message or email text for AI-powered phishing analysis.
  • URL Inspector — Dual-layer (heuristic + LLM) analysis of suspicious URLs.
  • Screenshot OCR Scanner — Upload images; Tesseract.js extracts text, which the LLM then classifies.
  • Offline Scan History — IndexedDB-backed history with search, filter, and CSV export.
  • Settings Panel — Model switching, theme switching, cache management, and factory reset.

Source File Breakdown

  • index.html — Full single-page app layout (~37 KB), all panels defined here.
  • src/app.js — Main controller (841 lines). Wires UI events, calls AI, renders results, manages history.
  • src/ai-service.js — Thin wrapper that creates the WebLLM engine in a Web Worker and exposes analyzeTextWithAI, analyzeUrlWithAI, extractTextFromImage.
  • src/ai-worker.js — The Web Worker host that boots MLCEngine and passes messages from the main thread.
  • src/style.css — Design system (~42 KB) with CSS variables for both Blossom Pink and Sage Mint themes.
  • vite.config.js — Vite + VitePWA config, service worker generation.

Dependency Stack

  • @mlc-ai/web-llm ^0.2.84 — WebGPU LLM runtime (Qwen2.5-0.5B).
  • tesseract.js ^7.0.0 — WASM OCR engine for image-to-text.
  • localforage ^1.10.0 — IndexedDB abstraction for offline history persistence.
  • lucide ^1.24.0 — Icon library, rendered via lucide.createIcons().
  • vite ^8.1.4 — Build tool & dev server.
  • vite-plugin-pwa ^1.3.0 — Service worker + manifest generation.

Complete Function Reference
Every major function explained with purpose, input/output, and code context
🚀
initAI()
ai-service.js · Async · Singleton pattern
AI Layer

Purpose: Boots the WebLLM engine inside a Web Worker. Uses a singleton guard so the model is loaded only once across the entire session. Fires on DOMContentLoaded in the background so the model is warm before the user submits their first scan.

How it works:

export async function initAI() {
  if (engine) return engine;
  engine = await CreateWebWorkerMLCEngine(
    new Worker('./ai-worker.js', { type: "module" }),
    "Qwen2.5-0.5B-Instruct-q4f16_1-MLC",
    { initProgressCallback }
  );
  return engine;
}
📝
analyzeTextWithAI(text)
ai-service.js · Async · Returns structured JSON risk report
Text Scanner

Purpose: Sends the user's pasted message/text to the LLM with a strict cybersecurity system prompt, parses the JSON response, and returns a structured risk object.

🔗
analyzeUrlWithAI(url, heuristics)
ai-service.js · Async · Merges heuristic signals with LLM output
URL Inspector

Purpose: Passes a URL plus pre-computed heuristic signals to the LLM. The model receives both the raw URL and the boolean heuristic flags as context, enabling much more accurate classification.

🔬
extractUrlHeuristics(url)
app.js · Synchronous · Rule-based Layer 1 scanner
Heuristics

Purpose: Layer 1 of the dual-layer inspection pipeline. Performs instant, zero-latency rule-based checks before the LLM is invoked. Returns a flags object covering HTTPS, typosquatting brand variations, high-risk TLDs, IP masks, and keywords.

📸
extractTextFromImage(fileOrDataUrl)
ai-service.js · Async · Tesseract.js WASM OCR
Screenshot Scanner

Purpose: Runs local optical character recognition on an uploaded image entirely in-browser using Tesseract.js compiled to WebAssembly.

Problems Faced & Solutions
Technical challenges encountered during development and how they were resolved
# Problem Root Cause Solution Implemented
1
Critical

UI freeze during LLM model loading 		WebLLM's model initialization runs GPU shader compilation on the main JavaScript thread, blocking all DOM events.
Solved

Moved all LLM operations into a Web Worker (ai-worker.js).
2
High

LLM returning markdown-wrapped JSON 		Qwen2.5 adds markdown code fences by default when asked to return structured data.
Solved

Added pre-parse sanitisation to strip fences before calling JSON.parse().
3
High

Tesseract OCR accuracy issues 		Garbled results on screenshots with mixed fonts or background graphics.
Solved

Implemented pre-computed templates as fallbacks for reliable testing.
4
Medium

GitHub README URL issue 		Cloning instructions pointed to an outdated organization repository.
Solved

Updated references to point to the correct personal repository path.
