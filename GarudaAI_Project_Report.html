Executive Overview
What is GarudaAI and why does it exist?

GarudaAI is a privacy-first, offline-first anti-phishing Progressive Web App (PWA) that detects phishing attempts entirely inside the user's browser — no servers, no cloud APIs, no telemetry. Every computation runs locally on the device's CPU/GPU.

🔒

Zero Telemetry

User inputs are never transmitted to any external server. All data stays inside the browser.

📴

Offline Capable

Once models are downloaded, GarudaAI can operate in fully air-gapped environments.

🤖

Local AI Inference

Uses WebGPU-accelerated LLM and WASM-compiled OCR for on-device intelligence.

The Problem

Traditional anti-phishing tools require users to upload sensitive content (messages, screenshots, links) to cloud servers — exposing private data, requiring internet connectivity, and adding latency. GarudaAI solves all three problems by moving every computation to the client.

Key Capabilities at a Glance

  • Text Scanner — Paste any suspicious message or email text for AI-powered phishing analysis.
  • URL Inspector — Dual-layer (heuristic + LLM) analysis of suspicious URLs.
  • Screenshot OCR Scanner — Upload images; Tesseract.js extracts text, which the LLM then classifies.
  • Offline Scan History — IndexedDB-backed history with search, filter, and CSV export.
  • Settings Panel — Model switching, theme switching, cache management, and factory reset.

Complete Function Reference
Every major function explained with purpose, input/output, and code context
🚀
initAI()
ai-service.js · Async · Singleton pattern
AI Layer

Purpose: Boots the WebLLM engine inside a Web Worker. Uses a singleton guard so the model is loaded only once across the entire session. Fires on DOMContentLoaded in the background so the model is warm before the user submits their first scan.

How it works:

export async function initAI() {
  if (engine) return engine; // singleton guard
  engine = await CreateWebWorkerMLCEngine(
    new Worker('./ai-worker.js', { type: "module" }),
    "Qwen2.5-0.5B-Instruct-q4f16_1-MLC",
    { initProgressCallback }
  );
  return engine;
}

Model: Qwen2.5-0.5B-Instruct-q4f16_1-MLC — a 4-bit quantized 500M-parameter model cached in browser Cache Storage after first download.

📝
analyzeTextWithAI(text)
ai-service.js · Async · Returns structured JSON risk report
Text Scanner

Purpose: Sends the user's pasted message/text to the LLM with a strict cybersecurity system prompt, parses the JSON response, and returns a structured risk object.

System Prompt enforces:

  • riskScore — integer 0–100
  • riskLevel — "Safe" | "Low" | "Medium" | "High" | "Critical"
  • confidence — e.g., "95.5%"
  • category — e.g., "Credential Theft", "Banking Scam"
  • foundKeywords — array of suspicious words found
  • evidence — technical explanation list
  • recommendation — actionable user advice

Has a fallback parser that strips markdown code fences (```json) before parsing, and returns a safe default if parsing still fails.

🔗
analyzeUrlWithAI(url, heuristics)
ai-service.js · Async · Merges heuristic signals with LLM output
URL Inspector

Purpose: Passes a URL plus pre-computed heuristic signals to the LLM. The model receives both the raw URL and the boolean heuristic flags (HTTPS, typosquatting, TLD, IP mask, etc.) as context, enabling much more accurate classification.

Return value: A merged object combining all heuristic boolean flags with the AI's riskScore, riskLevel, evidence, and recommendation.

🔬
extractUrlHeuristics(url)
app.js · Synchronous · Rule-based Layer 1 scanner
Heuristics

Purpose: Layer 1 of the dual-layer inspection pipeline. Performs instant, zero-latency rule-based checks before the LLM is invoked. Returns a flags object.

Checks performed:

  • 🔒 isHttps — Does URL use HTTPS?
  • ✂️ isShort — Is it a link shortener (bit.ly, tinyurl, t.co)?
  • 🌐 isIp — Is the hostname a raw IP address (regex: IPv4 pattern)?
  • 🎭 typosquattingFound — Does the hostname mimic PayPal, Google, Facebook, Netflix, Amazon, Apple, Microsoft, Chase, Wells Fargo, Bank of America? Includes homoglyph substitution detection (1→l, 0→o, 3→e).
  • ⚠️ isHighRiskTld — TLDs: .info .xyz .top .click .work .gq .cf .tk
  • 🔑 keywordsDetected — Path contains: login, signin, secure, account, verify, update, banking, support, recovery, free-gift
📸
extractTextFromImage(fileOrDataUrl)
ai-service.js · Async · Tesseract.js WASM OCR
Screenshot Scanner

Purpose: Runs local optical character recognition on an uploaded image entirely in-browser using Tesseract.js compiled to WebAssembly. No image data leaves the device.

export async function extractTextFromImage(fileOrDataUrl) {
  const worker = await Tesseract.createWorker('eng');
  const ret = await worker.recognize(fileOrDataUrl);
  await worker.terminate();
  return ret.data.text;
}

The extracted text is then passed directly to analyzeTextWithAI() for phishing classification.

📊
renderTextScanResults / renderUrlScanResults / renderScreenshotResults
app.js · Synchronous · UI rendering functions
UI

Purpose: Three parallel rendering functions — one per scan type — that take the structured JSON result object and update the DOM with:

  • Risk badge — coloured SUCCESS / WARNING / DANGER chip
  • Risk bar — animated progress bar with width = riskScore%
  • Highlighted text — suspicious keywords wrapped in <span class="suspicious-phrase-highlight">
  • Evidence list — each AI-returned evidence item as a bullet
  • Feature badges (URL only) — per-flag icons for HTTPS, typosquatting, TLD, IP, shortener, keywords
  • Suspicious pills (screenshot only) — found keyword chips
  • Recommendation box — changes icon between shield-check / shield-alert
📜
renderHistoryTable()
app.js · Synchronous · Filter + search + paginate scan log
History

Purpose: Re-renders the scan history table on every filter/search change. Reads from the in-memory scanHistory array (synced with IndexedDB via LocalForage).

Features:

  • Filter tabs: All / Text / URL / Screenshot
  • Live search by input text, category, or risk level
  • Per-row delete button with IndexedDB persistence
  • Formatted timestamps with locale-aware date/time
  • JSON export as a downloadable .json file (named with today's date)
💾
saveScanToHistory() & updateDashboardStats()
app.js · Persistence & Dashboard counters
Storage

saveScanToHistory(type, inputVal, result): Creates a timestamped record with a unique ID (Date.now().toString()), truncates long inputs to 80 chars for preview, prepends to the scanHistory array, and persists via localforage.setItem().

updateDashboardStats(): Recalculates total / safe / threat counts from the live scanHistory array and updates the three dashboard stat counters in real time after every scan.

🎨
Theme Switcher (Blossom Pink ↔ Sage Mint)
app.js · CSS Variable overrides at runtime
Settings

Purpose: Switches the entire visual palette at runtime without any page reload by overriding CSS custom properties on :root via JavaScript's setProperty().

Variables overridden: --color-pink-primary, --color-pink-accent, --color-pink-light, --color-pink-hover, --shadow-glow. Because the entire design system references these five variables, all UI components update simultaneously.

🖼️
drawMockOCRBoxes()
app.js · UI feedback during OCR pre-processing
UX

Purpose: Renders four randomised CSS-positioned bounding boxes overlaid on the loaded image to visually communicate that OCR scanning is happening. Each box is positioned with randomised top/left/width/height percentages to simulate real OCR region detection. This runs instantly on image load before the asynchronous OCR begins.

Problems Faced & Solutions
Technical challenges encountered during development and how they were resolved
# Problem Root Cause Solution Implemented
1
Critical

UI freeze during LLM model loading
The browser became completely unresponsive for 20–40 seconds when loading Qwen2.5-0.5B. 		WebLLM's model initialization runs GPU shader compilation on the main JavaScript thread, blocking all DOM events, animations, and user interactions.
Solved

Moved all LLM operations into a Web Worker (ai-worker.js) using CreateWebWorkerMLCEngine(). The worker runs in an isolated thread, keeping the main thread free for UI responsiveness. initProgressCallback was added to stream loading progress back to the UI badge.
2
High

LLM returning markdown-wrapped JSON
The model frequently output ```json\n{...}\n``` instead of raw JSON, causing JSON.parse() to throw. 		Qwen2.5, like most chat-tuned LLMs, adds markdown code fences by default when asked to return structured data, even when explicitly told not to.
Solved

Added a pre-parse sanitiser: rawContent.replace(/^```json\n?/, '').replace(/\n?```$/, '') before calling JSON.parse(). Added a full try/catch fallback that returns a safe default Medium-risk result if parsing still fails, preventing silent crashes.
3
High

Tesseract OCR unavailable for arbitrary user uploads
For uploaded images, Tesseract OCR gave inconsistent or garbled results on complex screenshots. 		Tesseract.js accuracy degrades on screenshots with mixed fonts, non-standard layouts, or images with heavy background graphics. Full OCR pipeline also required CORS-safe image handling.
Pragmatic

Implemented three built-in preset mockups (bank alert, delivery scam, safe invoice) with pre-computed OCR strings for reliable demo purposes. For user uploads, extractTextFromImage() runs Tesseract but the code also populates activeScreenshotText with a banking preset as a reliable fallback to showcase the full pipeline.
4
Medium

GitHub README pointing to a different repository
Badges and clone instructions referenced sachanarchita96-lgtm/GARUDA-SHIELD, confusing contributors. 		The project was migrated from a shared hackathon repository to the team lead's personal GitHub account, but the README was not updated before pushing.
Solved

Updated the Category badge, Track badge, and git clone command in README.md to reference aaryansingh66665/GarudaAI. Changes committed and pushed to GitHub.
5
Medium

Model size vs. accuracy trade-off
Smaller quantized models missed nuanced semantic phishing lures; larger models were too slow for browser use. 		Browser-side inference has tight memory constraints (typically 2–4 GB WebGPU limit) and no persistent GPU context like server-side inference.
Hybrid

Implemented a dual-layer approach: Layer 1 uses instant regex/rule-based heuristics for obvious signals (no GPU required). Layer 2 uses the quantized LLM only for semantic context. This allows fast first-pass filtering while reserving the slower LLM for ambiguous cases. Settings panel allows users to switch between 14.2 MB TinyBERT / 66.5 MB DistilBERT / 0 MB Heuristics-only modes.
6
Medium

Offline history not persisting across page reloads
Scan history was stored only in memory and was lost on refresh. 		JavaScript in-memory arrays are ephemeral — they reset when the browser tab is closed or refreshed.
Solved

Integrated LocalForage as an IndexedDB wrapper. Every scan result is persisted to the garuda_scan_history key immediately after analysis. On page load, the history is rehydrated from IndexedDB, making scan history persistent across sessions without any server or account.
7
Medium

PWA Install button non-functional
The "Install GarudaAI" button in Settings did not trigger the browser's PWA install prompt. 		Browser PWA install prompts require specific conditions (HTTPS, valid manifest, service worker) and must be triggered from within a beforeinstallprompt event handler, which was not implemented.
In Progress

Documented as an Upcoming Feature in the README. The service worker and manifest are already configured via vite-plugin-pwa; the beforeinstallprompt event listener and deferred prompt invocation will be added in the next release.
8
Low

Lucide icons not rendering after dynamic DOM updates
Icon elements added dynamically via innerHTML stayed as blank <i> tags. 		Lucide's icon replace loop runs once on page load. Dynamically injected data-lucide elements after that point are not processed.
Solved

Called lucide.createIcons() explicitly at the end of every render function (renderTextScanResults, renderUrlScanResults, renderScreenshotResults, renderHistoryTable) to re-scan and replace any newly added icon elements.

Scan Flows — Step by Step
How each scanner processes input from click to result

5.1 Text Scanner Flow

1
User pastes text → clicks "Analyze"

btnTextAnalyze.click triggers; validates non-empty; calls showResultsLoading() to display spinner.

2
AI inference

analyzeTextWithAI(text) → system prompt + user message → LLM chat completion → JSON parse → result object.

3
Persist + Render

saveScanToHistory('text', ...) → IndexedDB write. renderTextScanResults(result) → DOM update with badge, bar, highlights, evidence, recommendation.

4
Dashboard refresh

updateDashboardStats() recounts totals from updated history array.

5.2 URL Inspector Flow

1
User enters URL → clicks "Inspect"

Validation; showResultsLoading().

2
Layer 1 — Heuristic scan (synchronous, instant)

extractUrlHeuristics(url) → returns flags: isHttps, isShort, isIp, typosquattingFound, isHighRiskTld, keywordsDetected.

3
Layer 2 — LLM semantic analysis (async)

analyzeUrlWithAI(url, heuristics) → sends URL + heuristic flags as context → LLM returns JSON → merged result object.

4
Render per-feature badges

updateFeatureBadge() called 6 times for HTTPS, typosquatting, TLD, IP, shortener, keywords — each gets a coloured check/warning/X badge.

5.3 Screenshot Scanner Flow

1
User uploads image or selects preset mockup

FileReader API reads file as DataURL; or preset SVG image loaded directly. drawMockOCRBoxes() overlays scanning animation.

2
OCR extraction (for real uploads)

extractTextFromImage(dataUrl) → Tesseract.js WASM worker → returns extracted text string.

3
LLM classification

Extracted (or preset) text passed to analyzeTextWithAI(ocrText). Result tagged with result.ocrText for display.

4
Render screenshot results

Risk badge, bar, extracted text preview, suspicious keyword pills, recommendation box all updated by renderScreenshotResults().
